Intoto iOS

System Design

A backend-driven, offline-first, multi-tenant university engagement platform — built in Swift & UIKit on an MVVM architecture.

Swift · UIKit MVVM Realm · Offline-first Auth0 Socket.IO 6 roles · multi-campus
OVERVIEW· 02

What Intoto is

A single iOS app that carries an incoming student from invitation through arrival and campus life — and gives universities the roles to run it. The same binary serves six roles across many institutions, configured entirely by the backend.

6
User roles in one app — from Student to University Super Admin
N
Universities & campuses, each with its own forms, branding, and feature set
7
Localized languages, from English to Bengali, Hindi, French & Urdu
100%
Dashboards & forms driven by server config — no app release to change them
PRINCIPLES· 03

Design goals & constraints

Goal 01
Configure, don't ship
University requirements differ and change often. Forms, dashboard sections, and available features are described by the server so the client adapts without an App Store release.
Goal 02
Work without signal
Students fill long forms on the move. Edits persist locally on every step and sync in the background — connectivity is never a blocker.
Goal 03
One app, many tenants
Six roles and many institutions share one codebase. Identity, scope, and permissions decide what each session can see and do.
Goal 04
Modular by feature
Each capability is a self-contained MVVM module, so teams build and test in parallel against shared common services.

ARCHITECTURE· 06

Anatomy of a feature module

Every feature folder repeats the same shape, so any engineer can navigate an unfamiliar module on sight. Profile, Events, Community and Dashboard all follow it.

/View
Controllers, cells & XIBs that present the module and capture input.
/ViewModel
State, validation and the bindings the view observes.
/Model
Codable response & domain types — one per concern (e.g. BasicDetailsModel).
/Service
API client & data access for the feature, on top of shared networking.

Larger modules add helpers — RequestBuilder, Validator, DataAssembler, Routing — without breaking the convention.

CONFIGURATION· 08

Backend-driven configuration

The server describes the UI; the client renders it. Dashboards, forms and shortcuts arrive as data — so a university's onboarding can change without touching the app.

Dashboard
Sections in API order
Banner, Quick Actions, Events, Activities and Ambassadors render only when, and in the order, the server returns them.
Forms
formId → sections
studentForm and travelForm resolve to a section & field list — each field marked required, visible, editable.
Shortcuts
actionId routing
Quick-action tiles carry an actionId the client maps to a destination screen — config decides which appear.

Licensed features gate capabilities per tenant, so the same build exposes different surface area to different universities.

INFRASTRUCTURE· 09

The networking layer

A single async client over Alamofire: every request checks reachability, attaches a fresh Auth0 token, and maps the response through one parser before it reaches a service.

DATA· 10

Offline-first storage & sync

Form progress is written to Realm the moment a student taps Next, then pushed to the server by a background process. A dropped connection never stops the flow.

1 · Capture
User completes a section and taps Next.
→
2 · Persist local
Written to Realm immediately — source of truth offline.
→
3 · Sync background
Pushed to server when reachable; UI never blocks.
Realm
Heavy structured data — profiles, cached university & dashboard responses, migrations managed.
UserDefaults
Lightweight session state — tokens flags, selected role, language preference.
Read-through cache
University & student data cached on first fetch to avoid frequent re-calls.
SECURITY· 11

Authentication & document security

Identity · Auth0
Tokens & silent refresh
Credentials are obtained through Auth0's manager; the network client refreshes access tokens transparently and forces logout when a refresh token expires.
At rest
Encryption service
A dedicated encryption layer protects sensitive locally-stored values, separate from plain Realm records.
Document integrity
PDF tamper detection
Uploaded proofs and signed documents are checked for tampering before they're trusted in compliance flows.
Consent
Signature & terms
Travel sign-off captures a drawn or uploaded signature plus explicit Terms & Conditions consent before submission.
REAL-TIME· 12

Real-time chat

Messaging runs over a persistent Socket.IO connection — one-to-one and group threads, live unread counts, and presence — bridged to CallKit for voice.

ChatList
Inbox of threads with live unread counts.
ChatRoom
A conversation — history, send, real-time receive.
ChatGroup
Group threads tied to communities & cohorts.
UserList
Start a chat from any discovery list profile.

Socket events update threads in real time; CallKit integration surfaces voice calls through the native iOS call UI.

↓
